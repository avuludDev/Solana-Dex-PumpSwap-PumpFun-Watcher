# Solana Parser

Сервис для отслеживания активности на Solana mainnet: парсинг транзакций DEX-протоколов (PumpSwap, Meteora и др.), фильтрация по объёму, сохранение свапов и пулов в PostgreSQL, опционально — вебхуки Helius и Telegram-уведомления о «китах».

## Что делает проект

- **Подписка на сеть** — WebSocket `logsSubscribe` (Helius) по program id выбранных протоколов или **послотовый обход** через `getSlot` / `getBlock` с несколькими асинхронными воркерами.
- **Парсинг** — из сырых транзакций (`jsonParsed`) извлекаются события создания пулов и свапы; поддерживаются отдельные парсеры в каталоге `protocols/`.
- **Фильтры** — минимальная сумма свапа в USD, для части протоколов — порог по лампортам SOL на стороне парсера; композитные правила в `filters/`.
- **Оценка USD** — фоновое обновление курса SOL (Helius HTTP), пересчёт объёма свапа в долларах для прошедших фильтр событий.
- **Хранение** — батчевые репозитории `SwapRepository`, `PoolRepository`, `TokenRepository` в PostgreSQL (`storage/repositories.py`, `infra/db.py`). При недоступности БД раннер может работать только с логами.
- **Восстановление прогресса** — в режиме потоковой обработки слотов последний обработанный слот периодически пишется в `last_processed_slot.txt`, чтобы после рестарта продолжить с нужного места.

## Точки входа

| Скрипт / модуль | Назначение |
|-----------------|------------|
| `main.py` | Запуск `BotRunner.run()`: WebSocket-стрим по настройкам в `core/runner.py` (какие программы включены — см. список `streams` в коде). |
| `slot_indexer.py` | Режим **«слоты в реальном времени»**: старт с отставанием на N минут от текущего слота, очередь слотов, воркеры тянут блоки и гоняют тот же пайплайн, что и стрим. Логирует статистику и отставание от цепи. |
| `token_metadata_updater.py` | Фоновый цикл: находит в БД токены без `name` / `symbol` / `decimals`, подтягивает метаданные через Helius (DAS / `getAsset`), обновляет таблицу `tokens`. |
| `run_telegram_bot.py` | Асинхронный цикл бота: опрос БД и отправка в Telegram отфильтрованных сигналов о крупных покупках и «китах» (см. `bot/README.md` и переменные окружения `TELEGRAM_*`). |
| `python -m webhook_parser.server` | HTTP-сервер для вебхуков Helius: регистрация/поиск webhook по публичному URL из настроек, приём POST, разбор payload и вызов `BotRunner._handle_raw_tx`. |

Вспомогательно: `scripts/test_block.py` — проверка парсинга одного блока через `BotRunner.run_single_block`.

## Конфигурация

Основной конфиг — **`config/settings.py`**: RPC (WebSocket и HTTP), DSN PostgreSQL, пороги фильтров, параметры Telegram, `webhook_public_url`, идентификаторы программ Pump.fun / PumpSwap / Meteora.

Рекомендуется выносить секреты (токен бота, пароль БД, API-ключи RPC) в переменные окружения; для Telegram уже используются `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID` и др. (см. `TelegramConfig` в том же файле).

Файл **`settings.py` в корне** репозитория в текущем дереве импортов **не используется** — ориентируйтесь на `config/settings.py`.

## Зависимости и запуск

```bash
pip install -r requirements.txt
```

Зависимости: `asyncpg`, `websockets`, `aiohttp`, `base58`.

Примеры:

```bash
# WebSocket-режим (как настроено в BotRunner.run)
python main.py

# Послотовый «индексатор» с отставанием ~5 минут и 16 воркерами
python slot_indexer.py

# Дозаполнение метаданных токенов
python token_metadata_updater.py

# Telegram-бот
python run_telegram_bot.py

# Сервер вебхуков (нужен публичный HTTPS URL в настройках)
python -m webhook_parser.server
```

Для работы с БД должен быть доступен PostgreSQL с ожидаемой схемой таблиц (свапы, пулы, токены — по репозиториям в `storage/repositories.py`).

## Структура каталогов (кратко)

- `core/` — типы событий, `BotRunner` (оркестрация стрима, слотов, обработчиков).
- `infra/` — RPC-стрим, БД, логирование, оракул цен, клиент метаданных Helius.
- `protocols/` — парсеры протоколов.
- `filters/` — правила отбора пулов и свапов.
- `storage/` — доступ к данным в PostgreSQL.
- `bot/` — логика Telegram-уведомлений.
- `webhook_parser/` — приём и разбор вебхуков.
